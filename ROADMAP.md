# Roadmap

Rough notes on where this repo is going. Not a hard plan, no deadlines.
Check things off when done, rewrite or drop things that stop making sense.

## Done

- Imported `Documentation/` from torvalds/linux master, commit `8541d8f`.
- Four scripts under `scripts/` (init, sync, diff, translation-status).
  All use a blobless + shallow + sparse clone, so updating from upstream
  doesn't require pulling the full kernel tree.
- README, CONTRIBUTING, TRANSLATORS, and a stub `index.rst` under `vi_VN/`.
- `TRANSLATION_STATUS.md` is generated by a script, starts at 0%.
- Repo is public at https://github.com/tamnd/kernel-docs-vi.

## Phase 1: groundwork

Before translating a bunch of files, a few things need to be in place.
Otherwise we end up redoing work later.

### disclaimer-vi.rst

Every other translation (zh_CN, ja_JP, it_IT) ships a small disclaimer
that gets included at the top of each translated page. It basically says
"this is a translation, and if it disagrees with the original, the
original wins." Easiest path is to copy `zh_CN/disclaimer-zh_CN.rst` and
adapt the wording.

### GLOSSARY.md

Terminology needs to be settled early. Three groups to sort out:

1. Words we keep in English: kernel, userspace, process, thread, syscall,
   driver, module, scheduler, interrupt, RCU, spinlock, memory barrier,
   bootloader, BIOS. Vietnamese technical writing tends to keep these
   as-is, and forcing a translation usually reads badly.
2. Words we translate outright: system, file, directory, permission,
   error. These are unambiguous.
3. Words where either choice works, so we just pick one and stay
   consistent: lock, queue, buffer, pipe. There's no right answer here.
   Consistency matters more than the choice itself.

Format: three-column table (English, Vietnamese, notes). Grow it as we
go, not all at once upfront.

### First real file: process/howto.rst

This is the entry point to kernel development. If a Vietnamese reader
shows up and this one's missing, the whole translation looks empty. It's
long but not deeply technical, which makes it a good file to work out
format questions on: how to handle cross-references, what to do with
code blocks, where to leave English in place.

### admin-guide/README.rst and admin-guide/index.rst

Entry points for kernel users rather than developers. Short, practical.
Translating these alongside `howto.rst` means both audiences get
something from day one.

## Phase 2: coverage

Priority order, roughly by usefulness to Vietnamese readers:

1. `process/`. Coding style, submitting patches, email client config,
   maintainer handbook. This is what people look up when they want to
   contribute upstream.
2. Common pages under `admin-guide/`: kernel-parameters, bootloaders,
   module signing, sysctl, troubleshooting guides.
3. `core-api/`. Memory management, locking, concurrency primitives.
   Background knowledge that's useful even outside kernel work.
4. `dev-tools/`. kgdb, kasan, ftrace, perf. Practical, plenty of people
   need them.
5. `filesystems/`, `networking/`, `scheduler/`. Driven by contributor
   interest, not by alphabetical order.

No files-per-month target. A few solid translations a month is fine.
Bulk-translating to pad the count hurts more than it helps, because one
bad page makes readers doubt everything else.

## Phase 3: automation

Once we have a few dozen files translated, a few things become worth
building.

### Drift tracking

When upstream edits `process/howto.rst` after the Vietnamese version has
been merged, the translation starts drifting. We need a way to know: for
each translated file, what upstream SHA it was translated against, and
how far that has slipped since.

Simple approach: add a metadata line near the top of each translated
file.

```
:Upstream-at: 8541d8f725c6
```

A script walks the tree, compares each file's `Upstream-at` against the
current `UPSTREAM` commit, and lists files that need review. It doesn't
have to be exact. Early warning is enough.

### CI

GitHub Actions is enough, no need for anything heavier.

- Run `scripts/translation-status.sh` on every merge and every upstream
  sync. Commit the updated `TRANSLATION_STATUS.md` if it changed.
- Basic RST linting: docutils should be clean. A Sphinx build on top of
  that would be nicer but isn't strictly required yet.
- Check that translated files still have their SPDX header. Easy to lose
  in copy-paste, and it's a kernel-wide invariant.
- Check for a `:Original:` line pointing back at the source.

### Sphinx build

Try `make htmldocs` limited to `vi_VN/` to confirm our toctree builds.
With only `index.rst` there isn't much to test, but once there are a
dozen or so pages it's worth wiring up so build errors don't accumulate.

## Phase 4: community

Hardest part to predict. One person translating alone is slow and burns
out, so we need more people. A few practical moves:

- Write a project intro (context, goals, how to contribute) and post it
  on Vietnamese dev channels. Daynhauhoc, Facebook groups for Linux and
  open source, company forums in the kernel and embedded space.
- Tag short, easy files as `good first translation` in
  `TRANSLATION_STATUS.md` or a separate file. Standard way to onboard
  new contributors.
- Review pull requests within a few days. A PR sitting for two or three
  weeks tells a first-time contributor not to come back.
- Accept small PRs. One file per PR, no pressure to batch. Batches are
  harder to review and intimidating for new contributors.

Don't expect too much. Translating kernel docs has no money, no badge,
no pressure deadline. The people who stick around are the ones who
genuinely like this slice of the stack, and that's not a big crowd in
Vietnam.

## Things we're deliberately not doing

- No translating source code comments or docstrings. The source tree is
  upstream's turf.
- No variant translations. No adding our own examples, no restructuring,
  no slipping in personal commentary. Translations stay close to the
  original in meaning and paragraph layout. If the original reads badly,
  fix it upstream first, then translate.
- No parallel translations for multiple kernel versions. We track master
  HEAD and rebase on each sync. Anyone who needs an older version can
  check out an older commit.
- No bulk machine translation with manual cleanup. Machine output is
  fine as a reference for terminology or a tricky sentence, but don't
  paste it straight into a PR. Reviewers notice, and cleanup tends to
  eat whatever time it saved.
- No CLA. DCO is enough, same as the kernel itself.

## Long term

If this project is still alive after a couple of years, a plausible
outcome:

- 20 to 50 core files translated properly, concentrated in `process/`,
  `admin-guide/`, and `core-api/`.
- A small group of regular contributors, not dependent on any single
  person.
- A stable glossary that other Vietnamese technical translation projects
  can reference.
- Maybe a patch series to LKML to pull `Documentation/translations/vi_VN/`
  into the main kernel tree. That needs enough content to show the
  project is alive, not a one-shot drop.

Upstreaming isn't a required goal. A clean public mirror that tracks
upstream is already useful on its own.
